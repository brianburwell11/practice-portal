## Sandbox note

Alongside this work, the docs portal itself will host experiments — interactive widgets and sandboxed UI elements built against a fresh set of audio samples. Those samples let us prototype waveform rendering, drag-alignment interactions, and per-stem playback engines in isolation before wiring them into the admin wizard. Expect new MDX pages with React widgets landing next to this plan as the phases progress.

## Original request

> I have a bunch of stems to the same song, but some of them start a couple of seconds in to the recording. The new add song process should look like: upload stems (either from a folder or individual files), relabel and adds colors (keep the audio preview of the loudest couple seconds or whatever, but get rid of setting default volume and panning at this stage. also allow keeping mono vs stereo, this is also where you set the order of stems), then display the waveforms for each stem in their color, stacked vertically on top of each other. allow me to mute or solo individual waveforms. and then manually drag them into alignment. i should be able to drag positive or negative in time. when you transcode and upload to r2, include this offset so it's just a regular stem that's aligned. if i moved the stem later in time, add silence to the beginning. if i moved the stem to the left, trim the beginning.

## Context

Today every stem is assumed to begin at `t=0` with no way to compensate — the only way to correct misaligned sources is to pre-trim outside the app. The current add-song wizard also asks for default volume and pan during upload, which is premature: those are live mixing decisions, not ingest-time ones.

The goal: replace the single upload step with a two-stage flow — first **label/color/stereo/order**, then a new **alignment** stage where the user sees each stem's waveform stacked vertically, can mute/solo and play them together, and drags them horizontally until they line up. The drag offset is **baked into the audio during transcode** (silence prepended for positive offset, head trimmed for negative), so what lands on R2 is a set of aligned stems — no per-stem offset field in `SongConfig`, no runtime sync logic to add.

## Decisions

Captured during the planning conversation:

| Question | Decision | Why |
|---|---|---|
| Playback during alignment? | **Full transport + per-stem mute/solo** | Matches how stems will be used in the real player; reuses `AudioEngine` patterns. |
| Drag snapping? | **Freeform + keyboard nudge** (arrow = ~1ms, shift+arrow = ~10ms) | Pro-DAW feel; detection-based snapping fights the user. |
| Editing existing songs? | **Re-align requires re-upload** | Simpler — R2 stems are baked; no need to store originals. |
| `defaultVolume` / `defaultPan` in `StemConfig`? | **Keep with defaults** (0.7 / 0) | Least disruptive. Mixer-side edit of these values is a follow-up. |
| Zero-point on alignment canvas? | **Implicit t=0** — whatever is leftmost at save time | If every stem ends up right of zero, every stem gets silence padded at the head. |
| Stem end handling? | **Preserve native length** | Current player already tolerates differing durations. |
| When to compute peaks? | **Client-side after upload** | Reuse the `AudioBuffer` already decoded for preview / channel detection. |

## Architecture touchpoints

**Frontend — modify:**
- `src/admin/steps/StemsStep.tsx` — strip the Vol/Pan sliders; keep label, color, stereo, drag-reorder, preview.
- `src/admin/wizardReducer.ts` — bump `step` union to `1|2|3|4|5`, add `offsetSec` to `StemEntry`, add `SET_STEM_OFFSET` action.
- `src/admin/AdminWizard.tsx` — wire new step between stems and timing.
- `src/admin/steps/ReviewStep.tsx` — include offsets in the FormData POST as a JSON sidecar field.

**Frontend — new:**
- `src/admin/steps/AlignmentStep.tsx` — stacked waveform rows, per-row mute/solo, transport strip, pointer-drag + keyboard nudge.
- `src/admin/utils/stemPeaks.ts` — compute `Float32Array` min/max peaks from an `AudioBuffer`.
- `src/admin/utils/alignmentPlayback.ts` — lightweight per-stem source+gain engine; offset handled at scheduling time (no buffer re-padding on drag).

**Backend — modify:**
- `vite-plugin-config-api.ts` transcode endpoint — parse `offsets` sidecar; pad (`adelay`) for positive offset, trim (`-ss`) for negative, unchanged for zero.

**Schema — no change:**
- `stemConfigSchema` stays as-is — offsets are baked into the audio, not persisted.

## Phase 5: Bake offsets into transcode

- [ ] `ReviewStep.tsx`: build `offsets: Record<string, number>` keyed by `file.name`, append as `formData.append('offsets', JSON.stringify(offsets))`
- [ ] `vite-plugin-config-api.ts`: capture non-file `offsets` field in busboy (`on('field', ...)`)
- [ ] New helper `transcodeToOpusWithOffset(input, output, probe, loudness, offsetSec)` wrapping `transcodeToOpus`:
  - `offsetSec > 0`: `-af adelay={ms}|{ms},loudnorm=...` (adelay before loudnorm)
  - `offsetSec < 0`: `-ss {abs(offset)}` on input before filter chain; measure loudness on trimmed content
  - `offsetSec == 0`: unchanged path
- [ ] Preserve existing `fileMap` return shape

## Verification

1. **Typecheck + build**: `npm run build` (full tsc+vite, not just vite).
2. **Upload golden path**: `npm run dev`, add a song via `/admin` with 3 stems where one starts late.
   - Waveforms render in their colors, stacked.
   - Solo each one — only the soloed one is audible.
   - Play, hear misalignment. Drag the late stem left until transients line up. Replay — aligned.
   - Nudge with arrows / shift+arrows; watch offset readout update.
3. **Transcode verification**: complete upload. Inspect the resulting opus files:
   - Stem with `+1.5s` offset has 1.5s of silence at the start (ffprobe / visual).
   - Stem with `−0.8s` offset has its first 0.8s gone.
   - Stem with `0` offset is byte-similar to the pre-revamp output.
4. **Cross-check against main player**: uploaded song plays cleanly — all stems start together at `t=0` with no global `beatOffset` correction.
5. **Schema stability**: existing songs still load; `defaultVolume` / `defaultPan` still set to 0.7 / 0.
6. **Edit mode**: open an existing song's wizard; alignment step shows read-only state; "Replace stems" re-enters upload flow.

## Out of scope (follow-ups)

- Storing original (pre-transcode) files to allow non-destructive re-alignment.
- Transient / zero-crossing snapping.
- Mixer-side edit that writes `defaultVolume` / `defaultPan` back to `config.json`.
- Per-stem end padding/trim to a uniform song duration.
